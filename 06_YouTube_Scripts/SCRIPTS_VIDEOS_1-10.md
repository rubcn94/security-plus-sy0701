**[1:30 - 5:00] CONFIDENTIALITY (Confidencialidad)**

[Cambiar visual a candado]

"**Confidentiality = Solo las personas autorizadas pueden acceder a información.**

Imagine que tienes el historial médico de pacientes.

Confidentiality asegura que:
✅ Solo el doctor puede ver historial de SU paciente
✅ Recepcionista NO puede ver historiales
✅ Otros pacientes NO pueden ver tu historial

**Cómo se logra Confidentiality:**

[Mostrar en pantalla]

1. **Cifrado (Encryption)**
   - AES-256 para datos en reposo
   - TLS/SSL para datos en tránsito
   - Ejemplo: WhatsApp end-to-end encryption

2. **Control de acceso (Access Control)**
   - Permisos de archivos (chmod 600)
   - ACLs (Access Control Lists)
   - Autenticación multifactor (MFA)

3. **Data Loss Prevention (DLP)**
   - Prevenir que empleados envíen datos sensibles por email
   - Ejemplo: Gmail bloqueando números de tarjeta crédito

**Ejemplo REAL - Fallo de Confidentiality:**

[Mostrar noticia]

Equifax breach 2017:
- 147 millones de registros expuestos
- Nombres, SSN, fechas nacimiento, direcciones
- Causa: Vulnerabilidad no parcheada
- **Costo: $1.4 BILLONES** en multas/compensación

Confidentiality ROTA.

**Amenazas a Confidentiality:**

- Shoulder surfing (mirar sobre hombro)
- Dumpster diving (buscar en basura)
- Eavesdropping (escuchar conversaciones)
- Man-in-the-Middle attacks
- Malware (keyloggers, spyware)

**En el examen te preguntarán:**

'¿Qué control protege Confidentiality?'
- Respuesta correcta: Encryption, Access Control, MFA
- Respuesta INCORRECTA: Backups (eso es Availability), Hashing (eso es Integrity)"

**[5:00 - 9:00] INTEGRITY (Integridad)**

[Visual: Checksum]

"**Integrity = Los datos NO han sido modificados (accidental o maliciosamente).**

Ejemplo:

Descargas archivo del banco con tus transacciones.

Integrity asegura que:
✅ El archivo que descargaste es EXACTAMENTE el que el banco envió
✅ Nadie lo modificó en el camino
✅ No se corrompió durante descarga

**Cómo se logra Integrity:**

[Mostrar en pantalla]

1. **Hashing**
   - SHA-256, MD5 (deprecado)
   - Ejemplo: Descargar Ubuntu ISO + verificar hash

   ```
   Archivo: ubuntu-24.04.iso
   SHA-256: a1b2c3d4e5f6...
   ```

   Si cambias UN BYTE → hash completamente diferente

2. **Digital Signatures**
   - Firma con llave privada
   - Verifica con llave pública
   - Ejemplo: Firmar PDFs, emails (S/MIME)

3. **Checksums**
   - CRC (Cyclic Redundancy Check)
   - Detecta corrupciones accidentales

4. **Version Control**
   - Git: tracking cambios
   - Auditoría: quién cambió qué y cuándo

**Ejemplo REAL - Fallo de Integrity:**

[Mostrar diagrama]

SolarWinds hack 2020:
- Atacantes MODIFICARON update de software
- Inyectaron backdoor (SUNBURST)
- 18,000 clientes descargaron versión comprometida
- Nadie detectó hasta meses después
- **Causa: Falta de code signing verification**

Integrity ROTA.

**Amenazas a Integrity:**

- Unauthorized modifications (empleado malicioso cambia datos)
- Man-in-the-Middle (modificar datos en tránsito)
- Malware (ransomware cifra/corrompe archivos)
- Bit rot (corrupción física del disco)

**Diferencia Confidentiality vs Integrity:**

[Tabla en pantalla]

| Escenario | C o I? |
|-----------|--------|
| Alguien LEE tu password | Confidentiality rota |
| Alguien CAMBIA tu password | Integrity rota |
| Alguien DESCARGA base de datos | Confidentiality rota |
| Alguien MODIFICA base de datos | Integrity rota |

**En el examen:**

'¿Qué control protege Integrity?'
- Correcto: Hashing, Digital Signatures, Checksums
- Incorrecto: Encryption (eso es Confidentiality)"

**[9:00 - 13:00] AVAILABILITY (Disponibilidad)**

[Visual: Server uptime]

"**Availability = Los sistemas están accesibles cuando los necesitas.**

Ejemplo:

Intentas acceder a Gmail.

Availability asegura que:
✅ El servicio está ONLINE
✅ Responde en < 1 segundo
✅ Puedes enviar/recibir emails

**Cómo se logra Availability:**

[Mostrar en pantalla]

1. **Redundancia**
   - Múltiples servidores (load balancing)
   - Múltiples ISPs
   - Múltiples data centers (geo-redundancia)

2. **Backups**
   - 3-2-1 rule:
     * 3 copias de datos
     * 2 medios diferentes (disk + cloud)
     * 1 copia off-site

3. **RAID (Redundant Array of Independent Disks)**
   - RAID 1: Mirroring (2 discos idénticos)
   - RAID 5: Striping con paridad (tolera 1 fallo)
   - RAID 10: Mirror + Stripe (tolera múltiples fallos)

4. **Fault Tolerance**
   - UPS (Uninterruptible Power Supply)
   - Generadores
   - Clustering (múltiples servidores)

5. **Patch Management**
   - Mantener sistemas actualizados
   - Prevenir crashes por bugs

6. **DDoS Protection**
   - Cloudflare, Akamai
   - Rate limiting

**Métricas de Availability:**

[Mostrar tabla]

| Uptime | Downtime/año | Descripción |
|--------|--------------|-------------|
| 99% | 3.65 días | Malo |
| 99.9% | 8.76 horas | Aceptable |
| 99.99% | 52.56 minutos | Bueno |
| 99.999% | 5.26 minutos | Excelente (five nines) |

**Ejemplo REAL - Fallo de Availability:**

[Mostrar diagrama]

AWS S3 Outage 2017:
- Región US-EAST-1 caída 4 horas
- Afectó: Netflix, Airbnb, Spotify, miles de sitios
- Causa: Error humano (comando typo)
- **Costo: Millones en pérdidas**

**Amenazas a Availability:**

- DDoS attacks (Distributed Denial of Service)
- Hardware failures (disco duro falla)
- Natural disasters (incendio, inundación)
- Power outages
- Ransomware (cifra archivos, los hace inaccesibles)
- Sabotage (empleado malicioso apaga servidores)

**En el examen:**

'¿Qué control protege Availability?'
- Correcto: Backups, RAID, Redundancia, UPS, Clustering
- Incorrecto: Encryption, Hashing"

**[13:00 - 14:00] BALANCEANDO CIA**

[Volver a triángulo CIA]

"El truco de seguridad es BALANCEAR los 3.

A veces, mejorar uno empeora otro.

**Ejemplos:**

**Caso 1: Cifrado fuerte**
✅ Mejora Confidentiality (nadie lee datos)
❌ Puede reducir Availability (cifrar/descifrar toma tiempo, sistema más lento)

**Caso 2: Backups frecuentes**
✅ Mejora Availability (restaurar rápido)
❌ Puede afectar Confidentiality (más copias = más vectores de ataque)

**Caso 3: Control de acceso estricto**
✅ Mejora Confidentiality
❌ Puede reducir Availability (si es TAN estricto que usuarios legítimos no pueden acceder)

**Decisión de negocio:**

Security team dice: 'Necesitamos cifrar TODO'
Business team dice: 'Pero eso hace el sistema 30% más lento'

¿Qué hacer?

Balance:
- Cifrar datos MÁS sensibles (PII, tarjetas crédito)
- Datos menos sensibles (logs públicos) NO cifrar

**Risk management = Balancear CIA según prioridades del negocio.**"
